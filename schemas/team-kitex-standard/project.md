# project\.md（Kitex 微服务项目工程规约）

## 一、项目基本信息

- **项目类型**：Golang Kitex RPC 微服务项目

- **技术栈**：Go \+ Kitex RPC \+ Protobuf \+ GORM \+ Redis

- **规范体系**：OpenSpec 团队全局统一规范

- **架构模式**：五层分层架构（Handler / Service / Logic / DAL / Convert）

- **适用场景**：后端 RPC 微服务、内部服务通信、业务能力封装

## 二、工程整体约束

本项目严格遵循团队全局 Kitex 专属规范，自动适配 OpenSpec 框架识别机制，仅启用 RPC 架构规则，不兼容、不混入 Hertz Web 架构逻辑，所有开发、代码生成、提交校验必须对齐以下工程规约。

### 2\.1 核心工程原则

- 分层单一职责、依赖可控、禁止跨层非法调用

- 业务逻辑收敛至 Service / Logic 层，入口层极致轻量化

- 数据层统一分库隔离，杜绝多库模型混乱、裸 SQL 硬编码

- 全项目编码风格统一、注释统一、Lint 规则统一

- 所有可复用逻辑通用收敛，杜绝重复代码、碎片化逻辑

## 三、强制目录规约

项目目录结构固定，禁止私自新增、删除、篡改核心业务目录，所有业务代码必须归位：

- `/`根目录：仅服务启动、RPC 注册、入口 Handler

- `biz/service`：RPC 业务实现层，单接口单 Service 结构体

- `biz/logic`：全局无状态通用复用逻辑、工具、缓存规则

- `biz/dal`：数据访问层，严格按 `gormL/{db_name}/where` 分库管理

- `biz/convert`：模型双向转换层，仅做字段映射无业务逻辑

- `conf`：全局配置管理，禁止硬编码任意配置

- `idl`：Protobuf 接口定义，按业务领域模块化拆分

- `kitex_gen`：工具自动生成代码，**禁止手动修改**

- `client`：外部 RPC 客户端统一封装

## 四、分层依赖白名单（强制）

所有代码依赖必须严格遵循白名单，CI/OpenSpec 校验强制拦截非法依赖：

```plain
Handler → Service
Service → Logic
Service → DAL
Service → Convert
Logic   → DAL（仅只读查询）
Convert → Logic
```

## 五、分层职责规约

### 5\.1 Handler 层

仅做请求透传、日志埋点、参数接收、错误包装、响应返回，**禁止编写任何业务逻辑、数据查询、参数校验、事务处理**。

### 5\.2 Service 层

RPC 业务核心实现层，负责业务流程编排、参数校验、事务控制、调用 DAL/Logic/Convert，允许直接操作数据库与缓存，一对一实现 IDL 接口能力。

### 5\.3 Logic 层

无状态通用能力层，存放多服务复用逻辑、计算规则、缓存策略、通用工具，不依赖上层业务，仅可只读查询 DAL 数据。

### 5\.4 DAL 层

统一数据访问层，分库隔离 MySQL、Redis 数据源，通过 where 链式构造器实现类型安全查询，Repo 无状态、用完即释，禁止全局单例。

### 5\.5 Convert 层

仅负责 DB 模型与 Protobuf 模型双向转换，只做字段映射与数据格式化，禁止嵌入任何业务判断与流程逻辑。

## 六、编码强制规约

全项目 Golang 编码必须严格对齐团队统一标准，所有 Lint、OpenSpec 校验不通过禁止提交合并：

- 结构体必须聚合初始化，杜绝未初始化告警、空指针风险

- 变量赋值优先使用 `:=` 自动类型推导，禁止冗余类型声明

- 禁止内外作用域变量重名，杜绝变量遮蔽问题

- 无限循环统一使用 `for{}`，禁止魔法数字写法

- JSON、含双引号字符串统一使用反引号原生字符串，禁止转义符拼接

- main 根目录禁止定义全局业务状态变量，所有状态封装结构体私有成员

- 静态可优化类 Lint 告警，行尾统一使用 `// NOLINT` 抑制

- 所有注释统一使用 `//` 单行中文注释，禁止非规范注释格式

## 七、文档与语言规约

- 项目所有需求文档、设计文档、注释、业务描述 **强制纯中文**

- 禁止拼音命名、中英文混杂、无意义缩写、口语化注释

- 所有 OpenSpec 拆解、AI 代码生成、架构评审必须遵循本文档规约

## 八、代码提交与门禁规约

- 代码开发完成后必须自查 golint、staticcheck 所有告警并整改

- 必须通过 `openspec validate` 规范校验

- 分层非法依赖、目录不规范、编码不统一、文档不规范一律拦截

- 禁止手动修改 kitex\_gen 自动生成代码

## 九、通用能力规约

- 所有列表查询统一使用 limit\+1 分页逻辑，统一分页结构体与 HasMore 判定

- 所有数据库查询优先使用 where 链式构造器，杜绝硬编码 SQL

- 缓存操作统一封装至 DAL redis 层，业务层禁止直接操作 Redis

> （注：文档部分内容可能由 AI 生成）
