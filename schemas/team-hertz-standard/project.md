# project\.md（Kitex 微服务项目工程规约）

## 一、项目基本信息

- **项目类型**：Golang Kitex RPC 微服务项目

- **技术栈**：Go \+ Kitex RPC \+ Protobuf \+ GORM \+ Redis

- **规范体系**：OpenSpec 团队全局统一规范

- **架构模式**：五层分层架构（Handler / Service / Logic / DAL / Convert）

- **适用场景**：后端 RPC 微服务、内部服务通信、业务能力封装

## 二、工程整体约束

本项目严格遵循团队全局 Kitex 专属规范，自动适配 OpenSpec 框架识别机制，仅启用 RPC 架构规则，不兼容、不混入 Hertz Web 架构逻辑，所有开发、代码生成、提交校验必须对齐以下工程规约。

### 2\.1 核心工程原则

- 分层单一职责、依赖可控、禁止跨层非法调用

- 业务逻辑收敛至 Service / Logic 层，入口层极致轻量化

- 数据层统一分库隔离，杜绝多库模型混乱、裸 SQL 硬编码

- 全项目编码风格统一、注释统一、Lint 规则统一

- 所有可复用逻辑通用收敛，杜绝重复代码、碎片化逻辑

## 三、强制目录规约

项目目录结构固定，禁止私自新增、删除、篡改核心业务目录，所有业务代码必须归位：

- `/`根目录：仅服务启动、RPC 注册、入口 Handler

- `biz/service`：RPC 业务实现层，单接口单 Service 结构体

- `biz/logic`：全局无状态通用复用逻辑、工具、缓存规则

- `biz/dal`：数据访问层，严格按 `gormL/{db_name}/where` 分库管理

- `biz/convert`：模型双向转换层，仅做字段映射无业务逻辑

- `conf`：全局配置管理，禁止硬编码任意配置

- `idl`：Protobuf 接口定义，按业务领域模块化拆分

- `kitex_gen`：工具自动生成代码，**禁止手动修改**

- `client`：外部 RPC 客户端统一封装

## 四、分层依赖白名单（强制）

所有代码依赖必须严格遵循白名单，CI/OpenSpec 校验强制拦截非法依赖：

```plain
Handler → Service
Service → Logic
Service → DAL
Service → Convert
Logic   → DAL（仅只读查询）
Convert → Logic
```

## 五、分层职责规约

### 5\.1 Handler 层

仅做请求透传、日志埋点、参数接收、错误包装、响应返回，**禁止编写任何业务逻辑、数据查询、参数校验、事务处理**。

### 5\.2 Service 层

RPC 业务核心实现层，负责业务流程编排、参数校验、事务控制、调用 DAL/Logic/Convert，允许直接操作数据库与缓存，一对一实现 IDL 接口能力。

### 5\.3 Logic 层

无状态通用能力层，存放多服务复用逻辑、计算规则、缓存策略、通用工具，不依赖上层业务，仅可只读查询 DAL 数据。

### 5\.4 DAL 层

统一数据访问层，分库隔离 MySQL、Redis 数据源，通过 where 链式构造器实现类型安全查询，Repo 无状态、用完即释，禁止全局单例。
1. 规则说明
    - 所有表的 repo 必须实现 GetWhere() 方法
    - 方法返回对应表的 where 构造器实例
    - 禁止手动 new where 对象，统一通过 repo 获取
2. 固定写法
```go
// GetWhere 强制规范
func (r *XxxRepo) GetWhere() *where.XxxListWhere {
return &where.XxxListWhere{}
}
```
### 5\.5 Convert 层

仅负责 DB 模型与 Protobuf 模型双向转换，只做字段映射与数据格式化，禁止嵌入任何业务判断与流程逻辑。

## 六、编码强制规约

- 变量赋值优先使用 `:=` 自动类型推导，禁止冗余类型声明

- 禁止内外作用域变量重名，杜绝变量遮蔽问题

- 无限循环统一使用 `for{}`，禁止魔法数字写法

- JSON、含双引号字符串统一使用反引号原生字符串，禁止转义符拼接

- main 根目录禁止定义全局业务状态变量，所有状态封装结构体私有成员

- 静态可优化类 Lint 告警，行尾统一使用 `// NOLINT` 抑制

- 所有注释统一使用 `//` 单行中文注释，禁止非规范注释格式

## 十、DAL层 Where 构造器强制规约（AI 生成核心标准）

本项目所有数据库查询条件，**必须使用仿 Ent 框架风格链式查询构造器**，统一存放于 `biz/dal/gormL/{db_name}/where` 目录，为项目强制标准，AI 代码生成、人工开发必须严格遵守，禁止任何形式非标实现。

### 10\.1 目录生成规则

- 每一个业务数据库，必须独立生成 `gormL/where` 目录，不允许跨库共用、不允许统一目录存放

- 单表对应独立 Where 文件，命名规范：`表名_where.go`

- 所有查询条件逻辑收敛至 where 层，Repo、Service 层禁止手写 Where 条件字符串

### 10\.2 代码架构标准（复刻 Ent 核心设计）

所有 Where 构造器强制采用「基类\+子类」分层设计，完全对齐 Ent 链式调用体验、类型安全特性：

- **统一 BaseWhere 基类**：全局通用，封装条件存储、拼接、Build 核心方法，所有业务 Where 结构体统一继承

- **业务子类 Where**：单表独立结构体，继承 BaseWhere，仅实现当前表字段查询方法

- **纯链式调用**：所有条件方法返回自身指针`*XxxWhere`，支持无限链式拼接

- **完全类型安全**：禁止硬编码 SQL 字符串、禁止裸参数拼接，所有字段、条件、参数强类型约束

### 10\.3 强制代码规范

- 所有字段查询方法命名规范：字段名\+条件（Eq、Neq、Gt、Lt、In、Like、Between 等）

- 内部条件存储字段、工具方法私有化，外部仅暴露链式业务方法，禁止外部直接操作底层条件

- 统一 `Build() *gorm.DB` 方法，统一拼接所有条件、返回可链式调用的 DB 实例

- Repo 层仅调用 Where 构造器，不参与条件拼接，实现**查询条件与数据CURD完全解耦**

- 严格复刻 Ent 使用体验，无学习成本、风格统一，杜绝自定义非标链式写法

### 10\.4 标准调用示例（统一风格）

```go
// 标准 Ent 风格链式调用
list, err := repo.List(ctx,
    where.NewXxxWhere().
        Status(1).
        CreateTimeGt(startTime).
        TitleLike(keyword).
        IdIn(ids...),
)

```

## 十一、AI 代码生成专属约束

- 自动识别项目 Kitex 框架属性，仅生成 RPC 专属架构代码，不混入 Web 逻辑

- 自动分库生成 `gormL/{db_name}/where` 完整目录与 Ent 风格查询代码

- 自动遵循分层依赖、编码规范、注释规范、中文规范

- 生成代码必须通过 golint、staticcheck、openspec 全量校验
